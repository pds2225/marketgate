# Codex Working Rules (STRICT)

## 목표
- 가장 빠르게 실행 가능한 MVP를 만든다.
- 1기능 = 1작업 = 1검증 원칙 유지

---

## 절대 원칙
- 기존 구조 유지
- 최소 변경
- 동작 깨지지 않게 수정
- 테스트 먼저 작성 후 코드 수정
- 실패 테스트만 통과시키는 범위에서 수정

---

## 금지 사항
- 전체 리팩토링 금지
- 불필요한 파일 생성 금지
- 새 라이브러리 추가 금지
- API 응답 구조 변경 금지
- 기존 함수명 변경 금지
- unrelated 코드 수정 금지

---

## 작업 방식

### 1. 작업 전
- 수정 대상 파일 명시
- 변경 목적 1줄 요약

### 2. 작업 순서
1. 테스트 작성
2. 테스트 실행
3. 실패 확인
4. 최소 코드 수정
5. 재테스트

---

## 수정 범위 제한
- 한 번에 최대 2개 파일만 수정
- 200줄 이상 수정 금지

---

## 에러 처리 (필수)
- 입력값 None 처리
- 빈값 처리
- 타입 오류 방지
- 예외 발생 시 fallback 유지

---

## 상태 처리 (필수)
- empty state
- error state
- loading state

---

## 로그
- 핵심 로직에는 최소 logging 추가
- print 남발 금지

---

## 테스트 규칙
- 기존 테스트 절대 깨지면 안됨
- 새로운 테스트는 명확한 케이스만 추가
- edge case 1개 이상 포함

---

## 출력 형식 (강제)
1. 변경 요약 (1~3줄)
2. 수정 파일 목록
3. 변경 코드 (diff 형태)
4. 테스트 결과
5. 영향 범위
6. 다음 작업 1개

---

## 동시작업 규칙
- Claude 작업 파일 절대 건드리지 마라
- tools/, docs/는 Claude 전용
- core 로직은 Codex 전용

---

## 실패 시 대응
- 실패 시 전체 재작성 금지
- 가장 작은 단위로 롤백 후 재시도
